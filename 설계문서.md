# coupon-rush 설계문서

> **출발점**: 실무에서 비관적 락으로 동시성을 제어한 경험이 있다. 트래픽이 올라가면 어디서 꺾일까? 그 질문에서 시작했다.

---

## 1. 문제 정의

### 왜 선착순 쿠폰인가?

세 가지 이유다.

1. **현실성**: 커머스에서 가장 빈번한 고부하 이벤트다. 개발자라면 한 번쯤 다뤄본 문제라 공감도가 높다.
2. **로직이 단순하다**: "수량이 남으면 발급, 없으면 거절"이 전부다. 비즈니스 복잡도가 낮을수록 전략 간 성능 차이가 기술 역량의 차이로 순수하게 드러난다.
3. **전략 간 전환이 자연스럽다**: 각 전략의 한계가 명확하고, 그 한계가 다음 전략의 동기가 된다.

### 목표 수치

목적별로 두 가지 테스트를 수행한다. 상세 기준과 논의 과정은 [부하 테스트](docs/infra/load-test.md) 참고.

| 테스트 | 쿠폰 수 | RPS | Duration | 목적 |
|--------|---------|-----|----------|------|
| 정합성 검증 | 1,000개 | 500 | 30초 | 모든 전략에서 소진 완료 → 초과발급 0% 확인 |
| 성능 비교 | 10,000개 | 500 | 30초 | 전략 간 처리량, 응답속도, 병목 비교 |

| 항목 | 목표값 | 근거 |
|------|--------|------|
| 초과 발급률 | 0% | 정확성은 타협 불가 |
| p99 latency 목표 | < 500ms | 사용자가 "느리다"고 느끼는 임계값 |

**RPS 점진적 증가**

500 RPS에서 시작해 전략 간 차이가 나타날 때까지 올린다. 상한은 10,000 RPS.

- **시작**: 500 RPS — 베이스라인
- **증가**: 차이가 안 나는 전략 간에만 1,000 → 2,000 → 5,000 → 10,000 RPS로 증가
- **이미 한계가 드러난 전략은 재테스트 불필요**

이 접근이 "전략의 한계를 실측으로 확인한다"는 프로젝트 원칙에 부합한다.
목표 RPS를 미리 고정하면, 그 수치에 도달 가능한지 자체가 인프라 변수에 휘둘린다.
대신 "어디서 꺾이는가"를 찾으면, 인프라 스펙과 무관하게 전략 간 상대적 차이를 비교할 수 있다.

**인프라 스펙**: 초기 테스트는 t3.small 환경에서 수행한다. 전략 간 차이가 충분히 나타나지 않을 경우 인스턴스 스펙을 상향한다 (t3.small → m6i.large 등). Terraform으로 변경 후 재실험.

### 왜 단일 서비스인가

이 프로젝트의 목적은 동시성 전략 비교다. MSA를 추가하면 네트워크 레이턴시, 서비스 디스커버리 등 변수가 섞여 전략 간 순수 성능 차이를 측정할 수 없게 된다. 단일 서비스로 변수를 통제한다.

## 2. 동시성 제어 전략 — 깨달음의 여정

### 왜 이 3개 전략인가

3개 전략은 **인프라 계층을 한 단계씩 확장한 것**이라는 일관된 축 위에 있다.

| 계층 | Step | 역할 분담 |
|------|------|-----------|
| **DB** | Step 1 (Pessimistic Lock, Single UPDATE) | DB가 락 + 데이터 전부 처리 |
| **DB + Redis** | Step 2 (Redis Lock, Redis Counter) | Redis가 동시성 제어, DB가 데이터 저장 |
| **Redis + Queue + DB** | Step 3 (Kafka, Redis Streams) | Redis가 판단, Queue가 전달, DB가 저장 |

각 전략은 이전 전략의 한계가 동기가 된다.
Step 1에서 DB 직렬화와 DB 원자 연산의 상한을 모두 확인한다.
Step 2에서 동시성 제어를 Redis로 이동하고, 락(Redis Lock)과 원자 연산(Redis Counter)을 비교한다.
Step 3에서 남은 DB 동기 저장 병목을 메시지 큐로 비동기화한다.
한 단계씩 변수를 바꿔야 어디서 차이가 생겼는지 알 수 있다.
한계를 실측으로 확인하기 전에 다음으로 넘어가지 않는다.

> **왜 낙관적 락(Optimistic Lock)은 포함하지 않았는가?**
> 낙관적 락은 충돌 빈도가 낮은 도메인에서 유효한 전략이다.
> 하지만 선착순 쿠폰은 수량이 줄어들수록 거의 모든 요청이 충돌한다.
> retry storm이 구조적으로 발생하는 도메인이므로 전략 비교 대상에서 제외했다.

> **왜 Kafka Queue(순수 큐 방식)는 포함하지 않았는가?**
> 요청을 Kafka에 넣고 Consumer가 순서대로 처리하면 동시성 문제 자체가 사라진다.
> 하지만 사용자는 즉시 결과를 받을 수 없고, polling이나 SSE가 필요하다.
> 선착순 쿠폰처럼 즉시 응답이 기대되는 도메인에서는 UX 트레이드오프가 크다.
> Step 3의 하이브리드가 Kafka의 장점(비동기 저장, 메시지 내구성)을 취하면서도 즉시 응답을 유지한다.

---

### Step 1: Pessimistic Lock (비관적 락)

**질문**: 실무에서 써온 비관적 락이 대규모 트래픽에서도 버티는가?

```sql
SELECT * FROM coupons WHERE id = ? FOR UPDATE
-- 락 획득 → 수량 확인 → 차감 → 커밋
```

**가설**: 정확성은 보장될 것이다. 단, 트래픽이 올라갈수록 어딘가에서 꺾일 것이다.

**측정 항목**:
- TPS (초당 처리량)
- p99 latency
- DB active connection 수
- Lock wait 시간

**예상**:

트래픽이 올라가면 DB connection pool이 고갈될 것이다. 락을 기다리는 스레드가 쌓이고, 대기 중인 요청은 타임아웃될 수 있다. 커넥션 풀 크기를 늘려도 DB 자체의 Lock contention이 병목이 될 것이다.

**서브 퀘스트: Single UPDATE**

SELECT FOR UPDATE의 한계를 확인한 뒤, "DB 안에서 할 수 있는 최선"도 실측한다.

```sql
UPDATE coupons SET issued_quantity = issued_quantity + 1
WHERE id = ? AND issued_quantity < total_quantity
-- affected rows = 0 → 소진, 거절
```

SELECT FOR UPDATE가 lock을 잡고 → 로직 → UPDATE → COMMIT까지 보유하는 반면, Single UPDATE는 한 문장에서 끝나므로 lock 보유 시간이 훨씬 짧다. 개념적으로 Redis INCR과 동일한 "원자 연산"이지만 매체가 디스크(DB)다. DB 원자 연산의 상한을 측정하고, Step 2의 Redis Counter(Redis 원자 연산)와 비교하면 "같은 개념, 다른 매체"의 차이를 숫자로 확인할 수 있다.

### Step 2: DB + Redis (동시성 제어를 Redis로 이동)

#### Redis Distributed Lock (분산 락)

**질문**: 분산 락은 정말 동시성 문제의 해결책인가? 성능은 올라가더라도 정합성은 보장되는가?

```
SET coupon_lock:{couponId} {requestId} NX PX 3000
-- 획득 성공 → 수량 확인/차감 → DEL
-- 획득 실패 → 거절 또는 대기
```

**가설**: DB 락 대신 Redis 락을 쓰면 처리량은 올라갈 것이다. 그러나 lock timeout 내에 작업이 끝나지 않으면 정합성이 깨질 수 있다. 분산 락은 "성능 문제의 해결책"이 아니라 "락의 위치를 옮긴 것"이며, 새로운 문제(timeout, 정합성)를 동반한다.

**측정 항목**:
- Step 1 대비 TPS
- Lock 획득 대기 시간
- Lock timeout으로 인한 실패율
- **lock timeout 시 정합성 깨짐 여부** (의도적으로 재현)

**예상**:

처리량은 Step 1보다 올라갈 것이다. Redis가 인메모리로 락을 처리하므로 DB connection pool 고갈 문제는 완화된다. 하지만 두 가지 한계가 예상된다:

1. **직렬화 구조 불변**: 여전히 "Lock을 획득한 요청만 처리"이므로, 병목이 DB에서 Redis로 이동했을 뿐 처리량의 상한은 존재한다
2. **정합성 보장 불가**: lock timeout이 만료되면 두 클라이언트가 동시에 임계 구역에 진입할 수 있다. GC pause, 네트워크 지연 등으로 실무에서 충분히 발생 가능한 시나리오

#### Redis Atomic Counter (락 없는 원자 연산)

**질문**: Lock을 아예 없애면 처리량의 상한이 사라지는가?

```
INCR coupon:{couponId}:issued
-- 반환값 > total → DECR 후 거절
-- 반환값 <= total → 발급 성공, DB에 동기 저장
```

**가설**: INCR은 Redis가 싱글 스레드로 원자 처리하므로, Lock 없이도 수량 정확성이 보장된다.
Lock wait가 없으니 처리량이 크게 올라갈 것이다.

**측정 항목**:
- Redis Lock 대비 TPS
- p99 latency
- DB 쓰기 대기 시간 (발급 성공 건의 INSERT 소요)
- Redis 장애 시나리오에서 정합성

**예상**:

TPS는 Step 1~2를 크게 앞설 것이다. Lock이 없으니 병렬 처리가 극대화될 수 있다.
단, 새로운 병목이 예상된다:

1. 발급 성공 시 DB에 Issuance 레코드를 동기로 INSERT하므로, 트래픽이 높아지면 DB 쓰기가 병목이 될 수 있다
2. Redis 장애 시 `issued` 카운터가 유실되면 DB와 정합성이 어긋날 수 있다

### Step 3: Redis + Queue + DB (비동기 저장)

**질문**: Step 2에서 락 병목은 제거했지만, DB 동기 저장이 새로운 병목이 됐다.
DB 저장을 비동기로 분리하면 처리량이 더 올라가는가?

#### 왜 메시지 큐인가

비동기 저장 방식은 여러 가지가 있다:

| 방식 | 장점 | 한계 |
|------|------|------|
| @Async / Spring Event | 단순, 인프라 추가 없음 | 앱 크래시 시 메모리에서 유실 |
| Redis List + 스케줄러 | 기존 Redis 활용 | 배치 주기에 따른 지연, 실패 처리 복잡 |
| 메시지 큐 (Kafka, Redis Streams) | 내구성, Consumer Group, 재처리 보장 | 인프라/운영 복잡도 증가 |

쿠폰 발급 기록은 금전적 가치가 있는 데이터다. "발급 성공했는데 기록 없음"은 허용할 수 없으므로 유실 방지가 핵심이다. @Async는 구조적으로 유실 가능하고, 메시지 큐의 ack 기반 재처리가 이 요구사항에 부합한다.

#### 왜 Kafka인가

메시지 큐 후보는 Kafka와 Redis Streams 두 가지다.

| 기준 | Kafka | Redis Streams |
|------|-------|---------------|
| INCR↔발행 원자성 | 불가 (외부 시스템) | Lua 스크립트로 가능 |
| 장애 도메인 | Redis와 분리 | Redis에 카운터+큐 집중 |
| 내구성 | 디스크 복제 (ISR) | 메모리 기반, AOF 의존 |
| 실무 사례 | 여기어때·배민·카카오 등 표준 | 선착순 쿠폰 프로덕션 사례 미발견 |
| 인프라 | MSK 추가 필요 | 기존 Redis 활용 |

Redis Streams는 Lua로 INCR+XADD를 원자화할 수 있어 dual-write gap이 원천 제거되는 기술적 장점이 있다. 그러나 카운터와 큐가 같은 Redis에 집중되어 단일 장애점 리스크가 커지고, 선착순 쿠폰 도메인에서의 프로덕션 사례가 부족하다.

Kafka를 우선 구현하고, Kafka의 dual-write gap 한계를 확인한 뒤 Redis Streams를 대안으로 검토한다.

#### 구현: Kafka

```
INCR coupon:{couponId}:issued
-- 반환값 > total → DECR 후 거절
-- 반환값 <= total → Kafka에 발급 이벤트 produce → 즉시 응답
-- publish 실패 시 → DECR 보상 → 에러 반환
-- Consumer → DB에 Issuance 레코드 저장
```

**가설**: API 응답은 Redis INCR + Kafka produce만 기다리므로 극도로 빠를 것이다.
DB 저장은 Consumer가 자체 속도로 처리하므로 DB 부하가 평탄화될 것이다.

**측정 항목**:
- Step 2 대비 TPS, p99 antml:parameter>
</invoke>
- Kafka Consumer lag (DB 반영 지연)
- DB 쓰기 부하 (Step 2 대비)
- 장애 시나리오: publish 실패율, Consumer 실패 시 메시지 유실 여부

**예상**:

Step 2 대비 TPS가 추가로 올라갈 것이다. API가 DB 쓰기를 기다리지 않으므로
응답이 빨라지고, Kafka Consumer가 DB 쓰기를 순차 처리하므로 DB 부하가 평탄화될 수 있다.

단, 새로운 운영 복잡도가 예상된다:

1. **INCR↔publish gap**: Redis INCR 성공 후 Kafka publish 실패 시 DECR 보상. 앱 크래시 시 카운터 1 높게 남을 수 있음 (1개 덜 발급, 초과발급보다 안전)
2. Consumer 처리 실패 시 발급 기록이 누락될 수 있다 → 재시도 + DLQ 필요
3. Redis 카운터와 DB 레코드 수가 불일치할 수 있다 → 주기적 reconciliation 필요
4. 동일 메시지가 중복 소비될 수 있다 → Idempotency Key로 중복 방지

#### 대안 검토 예정: Redis Streams

Kafka의 dual-write gap 한계를 확인한 뒤, Redis Streams + Lua 원자화를 대안으로 테스트한다.

```lua
-- INCR + XADD를 단일 Lua 스크립트로 원자 실행
local current = redis.call('INCR', KEYS[1])
local total = tonumber(redis.call('GET', KEYS[2]))
if current > total then
    redis.call('DECR', KEYS[1])
    return -1
end
redis.call('XADD', KEYS[3], '*', 'couponId', ARGV[1], 'userId', ARGV[2])
return current
```

트레이드오프: dual-write gap 제거 vs 단일 장애점 리스크 증가. 부하 테스트 후 비교 기록.

## 3. 전략 요약

| Step | 전략 | 핵심 아이디어 | 주요 한계 | 적합한 상황 |
|------|------|--------------|-----------|------------|
| 1. DB | Pessimistic Lock / Single UPDATE | DB가 락 + 데이터 전부 처리 | DB connection 고갈 | 트래픽이 낮고 정확성이 중요 |
| 2. DB + Redis | Redis Lock | 락을 인메모리로 이동 | 병목 이동 + lock timeout 시 정합성 위험 | 분산 락의 한계를 검증하는 실험 |
| | Redis Counter | Lock 없는 원자 INCR | DB 동기 저장이 병목 | 고성능 필요, 즉시 응답 필요 |
| 3. Redis + Queue + DB | Kafka / Redis Streams | Redis 판단 + 메시지 큐 비동기 저장 | dual-write gap, 운영 복잡도 | 고성능 + 안정성 모두 필요 |
