# coupon-rush 설계문서

> **출발점**: 실무에서 비관적 락으로 동시성을 제어한 경험이 있다. 근데 5만 RPS에서도 버틸까? 그 질문에서 시작했다.

---

## 1. 문제 정의

### 왜 선착순 쿠폰인가?

세 가지 이유다.

1. **현실성**: 커머스에서 가장 빈번한 고부하 이벤트다. 면접관이 직접 다뤄본 문제라 공감도가 높다.
2. **로직이 단순하다**: "수량이 남으면 발급, 없으면 거절"이 전부다. 비즈니스 복잡도가 낮을수록 전략 간 성능 차이가 기술 역량의 차이로 순수하게 드러난다.
3. **전략 간 전환이 자연스럽다**: 각 전략의 한계가 명확하고, 그 한계가 다음 전략의 동기가 된다.

### 목표 수치

| 항목 | 목표값 | 근거 |
|------|--------|------|
| 쿠폰 수량 | 10,000개 | 1,000개는 테스트가 너무 빨리 끝난다. 의미 있는 경합이 발생하는 최소 임계값 |
| 목표 동시 요청 | 50,000 RPS | 실무 루센 트레일(MAU 5,000)과 비교했을 때 수백 배 규모. 전략 간 차이가 충분히 드러나는 압력 |
| 초과 발급률 | 0% | 정확성은 타협 불가 |
| p99 latency 목표 | < 500ms | 사용자가 "느리다"고 느끼는 임계값 |

---

## 2. 동시성 제어 전략 — 깨달음의 여정

각 전략은 이전 전략의 실측 한계에서 시작한다.
"왜 이 전략을 선택했는가"는 항상 이전 테스트 결과에서 나온다.

---

### Step 1: Pessimistic Lock (비관적 락)

**질문**: 실무에서 써온 비관적 락이 대규모 트래픽에서도 버티는가?

```sql
SELECT * FROM coupons WHERE id = ? FOR UPDATE
-- 락 획득 → 수량 확인 → 차감 → 커밋
```

**가설**: 정확성은 보장될 것이다. 단, 트래픽이 올라갈수록 어딘가에서 꺾일 것이다.

**측정 항목**:
- TPS (초당 처리량)
- p99 latency
- DB active connection 수
- Lock wait 시간

**예상 결과와 한계**:

트래픽이 올라가면 DB connection pool이 고갈된다. 락을 기다리는 스레드가 쌓이고, 대기 중인 요청은 타임아웃된다. 커넥션 풀 크기를 늘려도 DB 자체의 Lock contention이 병목이 된다.

**대안 검토** (실측 후 채울 것):
- [ ] connection pool 크기를 늘리면?
- [ ] DB 서버 스펙을 올리면?
→ 기각 사유:

**이 단계에서의 깨달음**:

> "비관적 락은 정확하다. 그런데 트래픽이 올라가면 DB 자체가 병목이 된다. 정확성을 유지하면서 처리량을 높이려면, Lock contention을 줄여야 한다."

---

### Step 2: Optimistic Lock (낙관적 락)

**질문**: Lock을 잡지 않고 충돌을 감지하면 contention이 줄어들까?

```sql
UPDATE coupons
SET issued = issued + 1, version = version + 1
WHERE id = ? AND version = ? AND issued < total
-- 0 rows updated → 충돌 → retry
```

**가설**: Lock을 잡지 않으니 대기가 없다. 충돌 시 retry하면 정확성도 보장된다.

**측정 항목**:
- retry 횟수 (트래픽별)
- 실질 TPS (retry 포함)
- 충돌률 (conflict rate)

**예상 결과와 한계**:

충돌률이 낮은 도메인에서는 낙관적 락이 유리하다. 하지만 선착순 쿠폰은 쿠폰 수량이 줄어들수록 충돌률이 급격히 올라간다. 수량이 10개 남은 시점에서는 거의 모든 요청이 충돌하고, retry storm이 발생해 오히려 비관적 락보다 성능이 나빠진다.

**대안 검토** (실측 후 채울 것):
- [ ] retry 횟수 상한을 낮추면?
- [ ] retry 간격에 backoff를 적용하면?
→ 기각 사유:

**이 단계에서의 깨달음**:

> "경합이 심한 도메인에서는 낙관적 락도 답이 아니다. Lock contention을 줄이려다 retry storm이라는 다른 문제를 만들었다. 근본 문제는 DB에 Lock 판단을 맡기는 것 자체다."

---

### Step 3: Redis Distributed Lock (분산 락)

**질문**: Lock 판단을 DB에서 인메모리(Redis)로 옮기면 처리량이 달라질까?

```
SET coupon_lock:{couponId} {requestId} NX PX 3000
-- 획득 성공 → 수량 확인/차감 → DEL
-- 획득 실패 → 거절 또는 대기
```

**가설**: DB 락 대신 Redis 락을 쓰면, DB 부담이 줄고 처리량이 올라갈 것이다.

**측정 항목**:
- Step 1 대비 TPS 변화
- Lock 획득 대기 시간
- Lock timeout으로 인한 실패율

**예상 결과와 한계**:

DB connection pool 고갈 문제는 해결된다. Redis는 훨씬 빠르게 Lock을 처리한다. 그러나 여전히 "Lock을 획득한 요청만 처리"라는 구조는 바뀌지 않는다. 결국 Lock 획득 대기 = 직렬화다. 병목이 DB에서 Redis로 이동했을 뿐, Lock이라는 구조가 처리량의 상한을 만든다.

**대안 검토** (실측 후 채울 것):
- [ ] Lock timeout을 줄여서 실패를 빠르게 반환하면?
- [ ] Redis 클러스터로 분산하면?
→ 기각 사유:

**이 단계에서의 깨달음**:

> "병목이 DB에서 Redis로 이동한 것뿐이다. Lock이라는 개념 자체가 순차 처리를 강제한다. 락을 빠르게 처리해도 한 번에 하나씩 처리하는 건 변하지 않는다."

---

### Step 4: Redis Atomic Counter (락 없는 원자 연산)

**질문**: Lock을 아예 없애면 처리량의 상한이 사라지는가?

```
INCR coupon:{couponId}:issued
-- 반환값 > total → DECR 후 거절
-- 반환값 <= total → 발급 성공, DB에 비동기 기록
```

**가설**: INCR은 Redis가 싱글 스레드로 원자 처리하므로, Lock 없이도 수량 정확성이 보장된다. Lock wait가 없으니 처리량이 폭발적으로 올라갈 것이다.

**측정 항목**:
- Step 3 대비 TPS
- p99 latency
- DB 비동기 반영 지연
- Redis 장애 시나리오에서 정합성

**예상 결과와 한계**:

TPS는 Step 1~3을 크게 앞선다. Lock이 없으니 병렬 처리가 극대화된다. 단, 새로운 문제가 생긴다:

1. Redis 장애 시 `issued` 카운터가 유실되면 DB와 정합성이 어긋난다
2. DB 비동기 반영 실패 시 발급 기록이 누락될 수 있다

→ 이를 위해 Redis ↔ DB 주기적 reconciliation job 추가

**대안 검토** (실측 후 채울 것):
- [ ] Redis AOF/RDB로 영속성을 높이면?
- [ ] Redis Sentinel/Cluster로 고가용성을 확보하면?
→ 기각 사유:

**이 단계에서의 깨달음**:

> "성능 문제는 해결됐다. 그러나 '성능이 좋은 만큼 인프라 장애에 더 취약해졌다'. Redis를 믿는 만큼 Redis가 죽었을 때의 시나리오를 준비해야 한다. 고성능과 고가용성은 트레이드오프다."

---

### Step 5: Kafka Queue (동시성 문제 자체를 제거)

**질문**: Step 1~4는 모두 "동시 요청을 어떻게 처리할까"의 변주였다. 발상을 바꿔서, 요청을 큐에 넣고 순서대로 처리하면 동시성 문제 자체가 사라지지 않을까?

```
발급 요청 → Kafka 토픽 produce
Consumer (단일 파티션) → 순서대로 발급 처리
사용자 → 발급 결과를 polling 또는 SSE로 수신
```

**가설**: 단일 파티션 Consumer가 순서대로 처리하면 경쟁 자체가 없다. 처리량은 Consumer를 늘려 조절한다.

**측정 항목**:
- Consumer lag (처리 지연)
- 발급 요청 → 결과 수신까지 end-to-end 시간
- 처리량 (Consumer 수에 따른 변화)

**예상 결과와 한계**:

정확성은 완벽히 보장된다. 경쟁이 없으니 초과 발급은 구조적으로 불가능하다. 처리량도 Consumer를 늘리면 선형으로 증가한다.

단, 결정적인 트레이드오프가 있다: **사용자는 즉시 결과를 받을 수 없다**. "쿠폰 발급 중입니다"라는 중간 상태가 필요하고, 사용자가 결과를 polling하거나 서버가 SSE로 push해야 한다.

**대안 검토** (실측 후 채울 것):
- [ ] Consumer 수를 늘리면 처리량이 선형으로 증가하는가?
- [ ] SSE vs polling — 사용자 경험 차이는?
→ 기각 사유:

**이 단계에서의 깨달음**:

> "동시성 문제의 해결책 중 하나는 동시성 자체를 없애는 것이다. 즉시 응답을 포기할 수 있는 도메인이라면, Kafka 큐가 가장 안전하고 확장 가능한 구조다. 선착순 쿠폰처럼 '당첨 여부를 조금 늦게 알아도 되는' 도메인에서는 유효한 선택이다."

## 3. 전략 요약

| 전략 | 핵심 아이디어 | 주요 한계 | 적합한 상황 |
|------|--------------|-----------|------------|
| Pessimistic Lock | FOR UPDATE로 직렬화 | DB connection 고갈 | 트래픽이 낮고 정확성이 중요 |
| Optimistic Lock | version으로 충돌 감지 | 경합 심하면 retry storm | 충돌 빈도가 낮은 도메인 |
| Redis Lock | 락을 인메모리로 이동 | 병목이 이동했을 뿐 | DB 부담을 줄이고 싶을 때 |
| Redis Counter | Lock 없는 원자 INCR | Redis 장애 시 정합성 | 고성능 필요, 즉시 응답 필요 |
| Kafka Queue | 큐로 동시성 제거 | 즉시 응답 불가 | 비동기 처리 가능한 도메인 |

## 4. 기술 스택

| 기술 | 선택 | 선택 근거 |
|------|------|-----------|
| 언어 | Java 21 | 실무 언어. Virtual Thread로 blocking I/O 처리 비용 감소 (Lock wait 전략 비교 시 스레드 풀 고갈 방지) |
| 프레임워크 | Spring Boot 4.x | 실무 스택. 설명 가능한 코드 |
| DB | MySQL | 토스 JD 명시 스택 |
| 캐시/락 | Redis (Redisson) | 분산 락, Lua script, 원자 연산. Redisson으로 분산 락 편의성 확보 |
| 메시징 | Kafka | 순서 보장, 내구성. 토스 JD 명시 스택 |
| 부하 테스트 | k6 | JS DSL로 시나리오 작성 용이. Grafana와 실시간 연동 |
| 모니터링 | Prometheus + Grafana | 오픈소스. 전략별 메트릭을 동일 대시보드에서 비교 |
| 인프라 | Docker Compose + AWS | Docker Compose는 로컬 개발/디버깅용. 부하 테스트는 AWS에서 수행해 전략 간 동일 조건 보장 |
| IaC | Terraform | AWS 리소스(EC2, RDS, ElastiCache, MSK) 프로비저닝. 테스트 환경 setup/teardown 자동화로 반복 실험 생산성 확보 |
| 통합 테스트 | Testcontainers | 실제 MySQL/Redis/Kafka 컨테이너. H2는 락 동작이 달라 동시성 버그 재현 불가 |

**MSA를 쓰지 않는 이유**: 이 프로젝트의 목적은 동시성 전략 비교다. MSA를 추가하면 네트워크 레이턴시, 서비스 디스커버리 등 변수가 섞여 전략 간 순수 성능 차이를 측정할 수 없게 된다. 단일 서비스로 변수를 통제한다.

## 5. 프로젝트 구조

```
coupon-rush/
├── src/main/java/com/couponrush/
│   ├── domain/
│   │   └── coupon/          # Coupon(root) + Issuance 애그리거트, 전략, 서비스, 컨트롤러, DTO
│   ├── global/              # 전역 설정, 예외 핸들러
│   └── common/              # 공통 유틸, 기반 클래스
├── k6/
│   ├── scenarios/           # 전략별 부하 테스트 스크립트
│   └── reports/             # 테스트 결과 (실측 후 채울 것)
├── terraform/               # AWS 인프라 (EC2, RDS, ElastiCache, MSK)
├── docker-compose.yml       # 로컬 개발/디버깅용 (MySQL, Redis, Kafka, Prometheus, Grafana)
└── docs/
    └── strategy-comparison.md  # 전략별 결과 비교 리포트
```

**전략 전환 방식**: 코드 변경 없이 설정 파일만 바꿔 동일 조건에서 비교.

```yaml
coupon:
  strategy: pessimistic  # pessimistic | optimistic | redis-lock | redis-counter | kafka
```

## 7. 부하 테스트

```javascript
// k6 시나리오 — 50,000 RPS 스파이크
// 주의: preAllocatedVUs는 전략별 응답 시간에 따라 조정 필요
// 비관적 락처럼 응답이 느린 전략은 VU 부족으로 요청이 드롭될 수 있음
// 실측 시 dropped_iterations 지표를 확인하고 maxVUs 조정
export const options = {
  scenarios: {
    spike: {
      executor: 'constant-arrival-rate',
      rate: 50000,
      timeUnit: '1s',
      duration: '30s',
      preAllocatedVUs: 1000,
      maxVUs: 5000,  // 전략별 응답 속도에 따라 조정
    },
  },
  thresholds: {
    http_req_failed: ['rate<0.01'],   // 오류율 1% 미만
    http_req_duration: ['p(99)<500'], // p99 500ms 미만
  },
};
```

**정확성 검증**:
1. 발급된 쿠폰 수 = DB issuance 레코드 수 = Redis counter 값 (삼중 일치 확인)
2. 동일 userId 중복 발급 없음
3. 총 발급량이 10,000 초과 없음

## 8. 모니터링 (Grafana 대시보드)

- **API**: RPS, p50/p95/p99 latency, 오류율
- **DB**: active connection, Lock wait 시간, 쿼리 실행 시간
- **Redis**: ops/sec, 메모리, connection 수
- **Kafka**: Consumer lag, 처리량 (Step 5)
- **JVM**: 스레드 수 (Virtual Thread 효과), GC pause

## 9. 구현 순서

각 Step은 **구현 → k6 테스트 → 결과 기록 → 대안 검토 → 한계 분석** 순으로 진행한다.
다음 Step은 이전 Step의 한계 분석 결과에서 시작한다.

1. 인프라 구성
   - Docker Compose: 로컬 개발/디버깅 환경 (MySQL, Redis, Kafka, Prometheus, Grafana)
   - Terraform + AWS: 부하 테스트 환경 (EC2, RDS, ElastiCache, MSK)
2. 기본 도메인 구현 (Coupon, Issuance 엔티티, 발급 API 뼈대)
3. **Step 1**: Pessimistic Lock → 테스트 → 대안 검토 → DB connection 고갈 확인
4. **Step 2**: Optimistic Lock → 테스트 → 대안 검토 → retry storm 확인
5. **Step 3**: Redis Lock → 테스트 → 대안 검토 → 병목 이동 확인
6. **Step 4**: Redis Counter → 테스트 → 대안 검토 → TPS 돌파 + 정합성 시나리오 검증
7. **Step 5**: Kafka Queue → 테스트 → 대안 검토 → end-to-end 지연 측정
8. `docs/strategy-comparison.md` 작성: 전략별 수치 비교 + 깨달음 정리
