# coupon-rush 설계문서

> **출발점**: 실무에서 비관적 락으로 동시성을 제어한 경험이 있다. 근데 5,000 RPS에서도 버틸까? 그 질문에서 시작했다.

---

## 1. 문제 정의

### 왜 선착순 쿠폰인가?

세 가지 이유다.

1. **현실성**: 커머스에서 가장 빈번한 고부하 이벤트다. 개발자라면 한 번쯤 다뤄본 문제라 공감도가 높다.
2. **로직이 단순하다**: "수량이 남으면 발급, 없으면 거절"이 전부다. 비즈니스 복잡도가 낮을수록 전략 간 성능 차이가 기술 역량의 차이로 순수하게 드러난다.
3. **전략 간 전환이 자연스럽다**: 각 전략의 한계가 명확하고, 그 한계가 다음 전략의 동기가 된다.

### 목표 수치

| 항목 | 목표값 | 근거 |
|------|--------|------|
| 쿠폰 수량 | 10,000개 | 1,000개는 테스트가 너무 빨리 끝난다. 의미 있는 경합이 발생하는 최소 임계값 |
| 목표 동시 요청 | 5,000 RPS | 아래 근거 참고 |
| 초과 발급률 | 0% | 정확성은 타협 불가 |
| p99 latency 목표 | < 500ms | 사용자가 "느리다"고 느끼는 임계값 |

**왜 5,000 RPS인가?**

5,000 RPS × 30초 테스트 = 총 150,000 요청. 쿠폰 10,000개 기준 **경쟁률 15:1**.

이 수치는 세 가지 근거로 선택했다:
- **실무 환경 반영**: 부하 테스트 인프라가 t3.micro(2 vCPU, 1GB, 기본 CPU 크레딧 10%)로, 실제 운영도 t3 인스턴스를 사용하는 경우가 많다. 테스트 환경이 실무와 유사해야 결과에 의미가 있다.
- **전략 차이 충분**: 비관적 락의 TPS 상한이 ~1,000~2,000으로 예상된다. 5,000 RPS면 DB 락 전략의 한계가 명확히 드러나면서도, Redis/Kafka 전략은 처리 가능한 구간이다.
- **1,000 RPS (경쟁률 3:1)**: 모든 전략이 비슷하게 처리한다. DB 락도 충분히 버티므로 비교 의미가 없다.

즉, "t3.micro 인프라에서 안정적으로 도달 가능하면서, DB 기반 전략의 한계가 드러나는 구간"이 핵심 기준이다. 실측 후 전략 간 구분이 불충분하면 RPS를 올린다.

## 2. 동시성 제어 전략 — 깨달음의 여정

### 왜 이 4개 전략인가

4개 전략은 **병목을 한 단계씩 상위 계층으로 밀어올린 것**이라는 일관된 축 위에 있다.

1. **DB 직렬화** (Pessimistic Lock) — DB가 직접 락을 잡아 순서를 보장
2. **인메모리 락** (Redis Lock) — 락 판단을 DB에서 Redis로 이동
3. **락 제거** (Redis Counter) — 원자 연산으로 락 자체를 없앰
4. **비동기 저장** (Redis Counter + Kafka) — DB 동기 저장 병목을 Kafka로 비동기화

각 전략은 이전 전략의 한계가 동기가 된다.
Steps 1~3은 락의 추상화 수준을 올려 처리량의 상한을 끌어올린다.
Step 3에서 락을 제거한 뒤에도 남는 병목(DB 동기 저장)을 Step 4에서 Kafka 비동기화로 해결한다.
한 단계씩 변수를 바꿔야 어디서 차이가 생겼는지 알 수 있다.
한계를 실측으로 확인하기 전에 다음으로 넘어가지 않는다.

> **왜 낙관적 락(Optimistic Lock)은 포함하지 않았는가?**
> 낙관적 락은 충돌 빈도가 낮은 도메인에서 유효한 전략이다.
> 하지만 선착순 쿠폰은 경쟁률 15:1로, 수량이 줄어들수록 거의 모든 요청이 충돌한다.
> retry storm이 구조적으로 발생하는 도메인이므로 전략 비교 대상에서 제외했다.

> **왜 Kafka Queue(순수 큐 방식)는 포함하지 않았는가?**
> 요청을 Kafka에 넣고 Consumer가 순서대로 처리하면 동시성 문제 자체가 사라진다.
> 하지만 사용자는 즉시 결과를 받을 수 없고, polling이나 SSE가 필요하다.
> 선착순 쿠폰처럼 즉시 응답이 기대되는 도메인에서는 UX 트레이드오프가 크다.
> Step 4의 하이브리드가 Kafka의 장점(비동기 저장, 메시지 내구성)을 취하면서도 즉시 응답을 유지한다.

### Step 1: Pessimistic Lock (비관적 락)

**질문**: 실무에서 써온 비관적 락이 대규모 트래픽에서도 버티는가?

```sql
SELECT * FROM coupons WHERE id = ? FOR UPDATE
-- 락 획득 → 수량 확인 → 차감 → 커밋
```

**가설**: 정확성은 보장될 것이다. 단, 트래픽이 올라갈수록 어딘가에서 꺾일 것이다.

**측정 항목**:
- TPS (초당 처리량)
- p99 latency
- DB active connection 수
- Lock wait 시간

**예상**:

트래픽이 올라가면 DB connection pool이 고갈될 것이다. 락을 기다리는 스레드가 쌓이고, 대기 중인 요청은 타임아웃될 수 있다. 커넥션 풀 크기를 늘려도 DB 자체의 Lock contention이 병목이 될 것이다.

### Step 2: Redis Distributed Lock (분산 락)

**질문**: Lock 판단을 DB에서 인메모리(Redis)로 옮기면 처리량이 달라질까?

```
SET coupon_lock:{couponId} {requestId} NX PX 3000
-- 획득 성공 → 수량 확인/차감 → DEL
-- 획득 실패 → 거절 또는 대기
```

**가설**: DB 락 대신 Redis 락을 쓰면, DB 부담이 줄고 처리량이 올라갈 것이다.

**측정 항목**:
- Step 1 대비 TPS
- Lock 획득 대기 시간
- Lock timeout으로 인한 실패율

**예상**:

DB connection pool 고갈 문제는 해결될 것이다. Redis는 훨씬 빠르게 Lock을 처리할 수 있다. 그러나 여전히 "Lock을 획득한 요청만 처리"라는 구조는 바뀌지 않는다. 결국 Lock 획득 대기 = 직렬화이므로, 병목이 DB에서 Redis로 이동했을 뿐 Lock이라는 구조가 처리량의 상한을 만들 수 있다.

### Step 3: Redis Atomic Counter (락 없는 원자 연산)

**질문**: Lock을 아예 없애면 처리량의 상한이 사라지는가?

```
INCR coupon:{couponId}:issued
-- 반환값 > total → DECR 후 거절
-- 반환값 <= total → 발급 성공, DB에 동기 저장
```

**가설**: INCR은 Redis가 싱글 스레드로 원자 처리하므로, Lock 없이도 수량 정확성이 보장된다.
Lock wait가 없으니 처리량이 크게 올라갈 것이다.

**측정 항목**:
- Step 2 대비 TPS
- p99 latency
- DB 쓰기 대기 시간 (발급 성공 건의 INSERT 소요)
- Redis 장애 시나리오에서 정합성

**예상**:

TPS는 Step 1~2를 크게 앞설 것이다. Lock이 없으니 병렬 처리가 극대화될 수 있다.
단, 새로운 병목이 예상된다:

1. 발급 성공 시 DB에 Issuance 레코드를 동기로 INSERT하므로, 트래픽이 높아지면 DB 쓰기가 병목이 될 수 있다
2. Redis 장애 시 `issued` 카운터가 유실되면 DB와 정합성이 어긋날 수 있다

### Step 4: Redis Counter + Kafka 비동기 저장 (실무 아키텍처)

**질문**: Step 3에서 락 병목은 제거했지만, DB 동기 저장이 새로운 병목이 됐다.
DB 저장을 비동기로 분리하면 처리량이 더 올라가는가?

```
INCR coupon:{couponId}:issued
-- 반환값 > total → DECR 후 거절
-- 반환값 <= total → Kafka에 발급 이벤트 produce → 즉시 응답
-- Consumer → DB에 Issuance 레코드 저장
```

**가설**: API 응답은 Redis INCR + Kafka produce만 기다리므로 극도로 빠를 것이다.
DB 저장은 Consumer가 자체 속도로 처리하므로 DB 부하가 평탄화될 것이다.

**측정 항목**:
- Step 3 대비 TPS, p99 latency
- Kafka Consumer lag (DB 반영 지연)
- DB 쓰기 부하 (Step 3 대비)
- 장애 시나리오: Consumer 실패 시 메시지 유실 여부

**예상**:

Step 3 대비 TPS가 추가로 올라갈 것이다. API가 DB 쓰기를 기다리지 않으므로
응답이 빨라지고, Kafka Consumer가 DB 쓰기를 순차 처리하므로 DB 부하가 평탄화될 수 있다.

단, 새로운 운영 복잡도가 예상된다:

1. Consumer 처리 실패 시 발급 기록이 누락될 수 있다 → 재시도 + DLQ 필요
2. Redis 카운터와 DB 레코드 수가 불일치할 수 있다 → 주기적 reconciliation 필요
3. 동일 메시지가 중복 소비될 수 있다 → Idempotency Key로 중복 방지

**대안 검토** (실측 후 채울 것):

- [ ] Redis 장애 시 Kafka만으로 fallback 가능한가?

## 3. 전략 요약

| 전략 | 핵심 아이디어 | 주요 한계 | 적합한 상황 |
|------|--------------|-----------|------------|
| Pessimistic Lock | FOR UPDATE로 직렬화 | DB connection 고갈 | 트래픽이 낮고 정확성이 중요 |
| Redis Lock | 락을 인메모리로 이동 | 병목이 이동했을 뿐 | DB 부담을 줄이고 싶을 때 |
| Redis Counter | Lock 없는 원자 INCR | DB 동기 저장이 병목 | 고성능 필요, 즉시 응답 필요 |
| Redis Counter + Kafka | Redis 판단 + Kafka 비동기 저장 | 운영 복잡도 (DLQ, reconciliation) | 고성능 + 안정성 모두 필요 |

## 4. 기술 스택

| 기술 | 선택 | 선택 근거 |
|------|------|-----------|
| 언어 | Java 21 | 실무 언어. Virtual Thread로 blocking I/O 처리 비용 감소 (Lock wait 전략 비교 시 스레드 풀 고갈 방지) |
| 프레임워크 | Spring Boot 4.x | 실무 스택. 전략 패턴과 프로퍼티 기반 런타임 전환에 적합 |
| DB | MySQL | 행 레벨 락(FOR UPDATE) 지원. 비관적 락 전략 구현에 적합 |
| 캐시/락 | Redis (Redisson) | 분산 락, Lua script, 원자 연산. Redisson으로 분산 락 편의성 확보 |
| 메시징 | Kafka | 파티션 단위 순서 보장과 메시지 내구성. 큐 기반 전략에 필요 |
| 부하 테스트 | k6 | JS DSL로 시나리오 작성 용이. Grafana와 실시간 연동 |
| 모니터링 | Prometheus + Grafana | 오픈소스. 전략별 메트릭을 동일 대시보드에서 비교 |
| 인프라 | Docker Compose + AWS | Docker Compose는 로컬 개발/디버깅용. 부하 테스트는 AWS에서 수행해 전략 간 동일 조건 보장 |
| IaC | Terraform | AWS 리소스(EC2, RDS, ElastiCache, MSK) 프로비저닝. 테스트 환경 setup/teardown 자동화로 반복 실험 생산성 확보 |
| 통합 테스트 | Testcontainers | 실제 MySQL/Redis/Kafka 컨테이너. H2는 락 동작이 달라 동시성 버그 재현 불가 |

**MSA를 쓰지 않는 이유**: 이 프로젝트의 목적은 동시성 전략 비교다. MSA를 추가하면 네트워크 레이턴시, 서비스 디스커버리 등 변수가 섞여 전략 간 순수 성능 차이를 측정할 수 없게 된다. 단일 서비스로 변수를 통제한다.

```
├── src/main/java/com/couponrush/

├── k6/
│   ├── scenarios/           # 전략별 부하 테스트 스크립트
│   └── reports/             # 테스트 결과 (실측 후 채울 것)
├── terraform/               # AWS 인프라 (EC2, RDS, ElastiCache, MSK)
├── docker-compose.yml       # 로컬 개발/디버깅용 (MySQL, Redis, Kafka, Prometheus, Grafana)
└── docs/
    └── strategy-comparison.md  # 전략별 결과 비교 리포트
```

**전략 전환 방식**: 코드 변경 없이 설정 파일만 바꿔 동일 조건에서 비교.

```yaml
coupon:
  strategy: pessimistic  # pessimistic | redis-lock | redis-counter | redis-kafka
```

## 7. 부하 테스트

```javascript
// k6 시나리오 — 5,000 RPS 스파이크
// 주의: preAllocatedVUs는 전략별 응답 시간에 따라 조정 필요
// 비관적 락처럼 응답이 느린 전략은 VU 부족으로 요청이 드롭될 수 있음
// 실측 시 dropped_iterations 지표를 확인하고 maxVUs 조정
export const options = {
  scenarios: {
    spike: {
      executor: 'constant-arrival-rate',
      rate: 5000,
      timeUnit: '1s',
      duration: '30s',
      preAllocatedVUs: 200,
      maxVUs: 1000,  // 전략별 응답 속도에 따라 조정
    },
  },
  thresholds: {
    http_req_failed: ['rate<0.01'],   // 오류율 1% 미만
    http_req_duration: ['p(99)<500'], // p99 500ms 미만
  },
};
```

**정확성 검증**:
1. 발급된 쿠폰 수 = DB issuance 레코드 수 = Redis counter 값 (삼중 일치 확인)
2. 동일 userId 중복 발급 없음
3. 총 발급량이 10,000 초과 없음

## 8. 모니터링 (Grafana 대시보드)

- **API**: RPS, p50/p95/p99 latency, 오류율
- **DB**: active connection, Lock wait 시간, 쿼리 실행 시간
- **Redis**: ops/sec, 메모리, connection 수
- **Kafka**: Consumer lag, 처리량 (Step 4)
- **JVM**: 스레드 수 (Virtual Thread 효과), GC pause

## 9. 구현 순서

각 Step은 **구현 → k6 테스트 → 결과 기록 → 대안 검토 → 한계 분석** 순으로 진행한다.
다음 Step은 이전 Step의 한계 분석 결과에서 시작한다.

1. 인프라 구성
   - Docker Compose: 로컬 개발/디버깅 환경 (MySQL, Redis, Kafka, Prometheus, Grafana)
   - Terraform + AWS: 부하 테스트 환경 (EC2, RDS, ElastiCache, MSK)
2. 기본 도메인 구현 (Coupon, Issuance 엔티티, 발급 API 뼈대)
3. **Step 1**: Pessimistic Lock → 테스트 → 대안 검토 → DB connection 고갈 확인
4. **Step 2**: Redis Lock → 테스트 → 대안 검토 → 병목 이동 확인
5. **Step 3**: Redis Counter → 테스트 → 대안 검토 → DB 동기 저장 병목 확인
6. **Step 4**: Redis Counter + Kafka → 테스트 → 대안 검토 → 운영 복잡도 vs 성능 분석
7. `docs/strategy-comparison.md` 작성: 전략별 수치 비교 + 깨달음 정리
8. `README.md` 작성: 최종 실측 결과 반영
