# 0005 - 부하 테스트를 고정 RPS가 아닌 점진적 증가 방식으로 수행

| 항목 | 내용 |
|------|------|
| 상태 | accepted |
| 날짜 | 2026-03-16 |
| 분류 | performance |
| 모듈 | 전체 |
| 기술 | k6, AWS |

## 배경 (Context)

부하 테스트 목표 RPS를 설정해야 했다. 처음에 50,000 → 5,000으로 하향했으나, t3.micro에서 Spring Boot 벤치마크가 ~400 RPS 수준이라는 점에서 고정 목표 자체가 인프라 변수에 휘둘리는 문제가 있었다.

## 검토한 선택지 (Options Considered)

1. 고정 RPS 목표 (50,000 / 5,000) — 인프라 스펙에 따라 도달 가능 여부가 달라져 설계와 실행이 불일치
2. 점진적 증가 (500 → 최대 10,000) — 각 전략이 꺾이는 지점을 찾는 방식. 인프라 스펙과 무관하게 전략 간 상대적 차이를 비교 가능

## 결정 (Decision)

500 RPS에서 시작해 전략 간 차이가 나타날 때까지 점진적으로 올린다. 상한 10,000 RPS.
차이가 안 나면 인스턴스 스펙을 올린다 (Terraform으로 변경 후 재실험).

## 구현 요약 (Implementation Summary)

- `k6/scenarios/spike.js`에서 rate를 환경변수(`__ENV.RATE || '500'`)로 변경
- `k6 run -e RATE=2000 k6/scenarios/spike.js` 형태로 실행
- `maxVUs: 2000`으로 10,000 RPS까지 커버 가능하도록 여유 확보

## 결과 (Consequences)

- "어디서 꺾이는가"를 찾는 방식이라 인프라 스펙에 종속되지 않음
- 전략별 한계 RPS를 실측 데이터로 비교 가능
- 프로젝트 원칙("실측 기반 판단")과 부합
