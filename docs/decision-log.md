# Decision Log

이 문서는 기본 기술 선택의 이유를 남긴다.
형식: 날짜 / 결정 / 근거 / 대안 및 비선택 사유.

## 2026-03-05

### [결정] 로컬 의존 서비스는 `docker compose`로 관리한다
- 근거: MySQL/Redis/Kafka를 동일 명령으로 올리고 내릴 수 있어 온보딩과 재현성이 가장 좋다.
- 대안 및 비선택 사유:
  - 수동 설치: 팀원/환경마다 편차가 커서 재현성이 떨어진다.
  - Testcontainers only: 테스트에는 유효하지만 개발 중 상시 서비스 확인 용도로는 compose가 더 직관적이다.

### [결정] 로컬 MySQL 기본 버전은 `8.4`를 사용한다
- 근거: 8.4는 LTS 라인으로 안정성과 장기 유지보수 관점에서 유리하다.
- 대안 및 비선택 사유:
  - MySQL 9.x: 최신 기능은 있으나 초기 프로젝트 기준으로는 안정성보다 이점이 작다.
  - MySQL 8.0 latest patch: 사용 가능하지만 장기 지원 명확성에서 8.4 LTS가 우선이다.

### [결정] 설정 파일 기본 형식은 `application.yml`을 사용한다
- 근거: 계층형 설정(`spring.datasource`, `spring.jpa`, `management`, `coupon.*`)을 읽기 쉽게 유지할 수 있다.
- 대안 및 비선택 사유:
  - `application.properties`: 단순 설정에는 충분하나 설정이 늘어날수록 가독성이 떨어질 수 있다.

### [결정] Java package는 underscore 없이 소문자 도메인 규칙(`com.couponrush`)을 따른다
- 근거: Java 표준 관례와 도구/라이브러리 생태계의 기본 가정에 부합한다.
- 대안 및 비선택 사유:
  - `com.coupon_rush`: 문법상 가능하지만 관례 위반으로 유지보수 시 혼란이 생긴다.

### [결정] 첫 구현 단위는 `Coupon` 생성/조회의 수직 슬라이스(API+JPA)로 시작한다
- 근거: 동시성 전략 구현 전에 `요청 -> API -> JPA -> MySQL` 기본 흐름이 먼저 검증되어야 이후 문제를 분리해 학습하기 쉽다.
- 대안 및 비선택 사유:
  - 전략 패턴부터 구현: 초기 진입 복잡도가 높아 학습 초반 디버깅 포인트가 과도하게 늘어난다.

### [결정] 로컬 MySQL host port를 `3306`이 아닌 `13306`으로 고정한다
- 근거: 로컬 환경의 기존 MySQL(또는 충돌 가능 경로)과 분리해 앱이 의도한 compose MySQL로만 접속하도록 보장한다.
- 대안 및 비선택 사유:
  - `3306` 유지: 환경에 따라 이미 사용 중일 수 있고, 충돌 시 원인 추적 비용이 커진다.

### [결정] DB 문자셋/콜레이션/시간대는 서버(`docker-compose`)에서 관리하고 앱 JDBC 옵션은 최소화한다
- 근거: 설정 책임을 한 곳으로 모아 중복/불일치 위험을 줄이고, 애플리케이션 설정을 단순하게 유지할 수 있다.
- 대안 및 비선택 사유:
  - 앱 URL 파라미터(`serverTimezone`, `characterEncoding`)도 함께 유지: 즉시 동작은 가능하나 서버 설정과 중복되어 장기적으로 관리 포인트가 늘어난다.

### [결정] 컨트롤러는 입출력만 담당하고 쿠폰 비즈니스 로직은 `CouponService`로 분리한다
- 근거: HTTP 계층과 도메인 로직의 책임을 분리하면 테스트 대상이 명확해지고 이후 전략 확장 시 변경 영향 범위를 줄일 수 있다.
- 대안 및 비선택 사유:
  - 컨트롤러에서 레포지토리 직접 호출 유지: 초기 구현은 빠르지만 로직이 쌓일수록 컨트롤러가 비대해지고 재사용/테스트가 어려워진다.
