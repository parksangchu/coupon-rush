# coupon-rush 방향 보정 제안서

- 작성일: 2026-03-17
- 독자: Claude와 공동 작업 중인 현재 프로젝트 소유자
- 목적: 현재 프로젝트 방향이 맞는지 점검하고, 다음 구현/문서화 우선순위를 정렬하기 위한 제안

---

## 한줄 결론

`coupon-rush`의 큰 방향은 맞다.  
실무에서 경험한 비관적 락 기반 동시성 제어를 출발점으로 삼아, 더 높은 처리량을 위한 전략을 단계적으로 비교하는 구조는 이력서/면접용 프로젝트로 충분히 강하다.

다만 현재 상태에서 그대로 확장하면 `전략 비교 프로젝트`로서의 공정성과 문서 초점이 흐려질 수 있다.  
다음 단계는 기능 추가보다 아래 3가지를 먼저 정교화하는 것이 좋다.

- 비교 실험의 공정성
- 전략 간 공통 계약 유지
- 문서화 초점 정리

---

## 현재 방향의 강점

이 프로젝트가 좋은 이유는 단순히 기술을 많이 붙였기 때문이 아니다. 아래 흐름이 일관되기 때문이다.

- 실무 경험에서 출발한다
  - `trail`에서 비관적 락으로 동시성을 제어한 경험이 출발점이다.
- 문제 정의가 선명하다
  - 선착순 쿠폰은 고부하 + 정합성 + 전략 비교가 모두 자연스럽게 드러나는 주제다.
- 전략 전환의 이유가 연결된다
  - DB 락 -> Redis 락 -> Redis 원자 연산 -> Kafka 비동기 저장으로 병목을 한 단계씩 밀어올리는 축이 명확하다.
- 실측 기반 판단을 목표로 한다
  - 추상적인 장단점 정리가 아니라 k6, 모니터링, Terraform을 통해 실제 측정하려는 방향이다.
- 단일 서비스로 변수를 통제한다
  - MSA를 억지로 도입하지 않고 전략 비교에 필요한 최소 변수만 남긴 점이 좋다.

즉, 이 프로젝트는 `토이 프로젝트`보다는 `실무 문제를 확장 검증하는 실험 프로젝트`로 읽힌다. 이 축은 유지해야 한다.

---

## 지금 보정이 필요한 이유

현재 구조는 좋지만, 그대로 확장하면 아래 문제가 생길 수 있다.

### 1. 부하 테스트 지표가 왜곡될 수 있다

현재 k6 시나리오는 일정 시간 동안 쿠폰 발급을 계속 때리는 구조다.  
그런데 쿠폰 수량이 적고 요청이 많으면, 실제 발급 성공 구간보다 `소진 후 빠르게 실패하는 요청` 비중이 훨씬 커질 수 있다.

그러면 아래 같은 왜곡이 생긴다.

- `p99 latency`가 발급 성공 경로보다 거절 경로 위주로 측정될 수 있음
- 전략 간 진짜 차이보다 `소진 이후 응답 특성`이 더 크게 반영될 수 있음
- "정말 이 전략이 빨랐는가?"보다 "빨리 409를 줬는가?"가 결과를 좌우할 수 있음

즉, 지금부터는 `발급 성공 성능 비교`와 `소진 후 정합성 확인`을 분리해서 보는 방향이 필요하다.

### 2. Step 4에서 전략 간 계약이 깨질 위험이 있다

현재 도메인/응답 구조는 발급 즉시 `Issuance`를 반환하는 흐름에 맞춰져 있다.  
그런데 Step 4는 Redis 판단 후 Kafka에 이벤트를 넣고 즉시 응답하는 비동기 저장 모델이다.

이때 문제가 생긴다.

- DB 저장 전에는 `issuanceId`가 아직 없을 수 있다
- Step 1~3과 Step 4의 API 의미가 달라질 수 있다
- 결과적으로 "같은 API를 다른 전략으로 비교"하는 프로젝트가 아니라, 중간에 다른 프로젝트가 섞인 것처럼 보일 수 있다

이건 나중에 생각할 문제가 아니라, Step 2로 넘어가기 전에 기준을 정해두는 것이 좋다.

### 3. 문서화가 열심히 되고 있지만 핵심에 더 집중할 필요가 있다

ADR과 문서화 습관 자체는 매우 좋다.  
다만 지금 단계에서 일부 문서는 프로젝트 고유의 판단보다 일반적인 개발 규칙에 가깝다.

예:

- package convention
- controller/service 분리
- 로컬 개발 환경 일반 설정

이런 문서도 나쁘지는 않다. 하지만 `coupon-rush`가 강해지는 건 여기서가 아니다.  
프로젝트의 설득력은 아래 문서에서 생긴다.

- 왜 이 4개 전략인가
- 어떤 지표로 비교할 것인가
- 정합성을 어떻게 검증할 것인가
- 어느 지점에서 병목이 이동하는가
- 빠른 전략이 왜 운영 복잡도를 높이는가

즉, 문서화 양이 아니라 `문서화 초점`을 조정할 시점이다.

---

## 유지해야 할 원칙

다음 원칙은 계속 유지하는 것이 맞다.

- 4개 전략 비교 축 유지
- 실측 기반 판단 유지
- 정합성 우선
- 단일 서비스 구조 유지
- 점진적 RPS 증가 방식 유지
- Step별로 한계를 확인하고 다음 단계로 이동

이 원칙이 흔들리면 프로젝트가 `기술 나열`로 보일 수 있다.

## 제안하는 보정 방향

### 1. 부하 테스트를 목적별로 분리

하나의 시나리오로 모든 것을 보려 하지 말고 목적을 분리하는 것이 좋다.

권장 분리:

- `성공 구간 성능 비교`
  - 발급 성공이 충분히 발생하는 구간에서 TPS/p99 측정
- `소진 구간 정합성 확인`
  - 초과 발급 없음, 중복 발급 방지, 실패율 확인
- `한계 구간 탐색`
  - 점진적 RPS 증가로 어디서 꺾이는지 기록

핵심은 `빠르다`와 `정확하다`를 같은 그래프에 억지로 우겨넣지 않는 것이다.

### 2. Step 4까지 고려한 공통 계약을 먼저 정하기

지금부터 정해야 할 질문은 이것이다.

> 이 프로젝트는 "전략만 바뀌고 API 의미는 같은 비교 프로젝트"인가,  
> 아니면 Step 4에서 아예 다른 응답 모델을 허용하는 프로젝트인가?

권장 방향은 전자다.

- 가능하면 전략이 바뀌어도 외부 API 의미는 최대한 유지
- Step 4도 즉시 응답 모델 안에서 비교 가능하도록 계약을 재설계
- 정말 다르게 가야 한다면, Step 4는 별도 비교 축으로 문서에 명확히 분리

중요한 건 `나중에 설명하기 쉬운 구조`다.

### 3. 문서화의 중심을 "실험과 의사결정"으로 이동

앞으로 추가되는 문서는 아래 중심으로 가는 것이 좋다.

- 전략 선택 이유
- 측정 지표 정의
- 테스트 시나리오 설계 근거
- 정합성 검증 방식
- 전략별 장단점과 운영 복잡도
- 결과 해석

즉, 문서는 "이 프로젝트에서만 의미 있는 판단"을 더 많이 담아야 한다.

### 4. 결과 문서를 미리 표준화

각 Step이 끝날 때마다 같은 형식으로 기록할 수 있어야 비교가 쉬워진다.

권장 템플릿 항목:

- 전략 이름
- 핵심 아이디어
- 테스트 환경
- 측정 시나리오
- TPS
- p99 latency
- 정합성 결과
- 병목 지점
- 운영상 복잡도
- 다음 단계로 넘어가는 이유

이 템플릿이 있어야 나중에 `전략 비교표`와 `이력서 문장`이 자연스럽게 나온다.

## 이 프로젝트에서 써야 할 어려움의 방향

이 프로젝트의 어려움은 `전략 외 이야기`로 따로 만들 필요가 없다.  
오히려 전략을 공정하게 비교하려다 생긴 문제를 어려움으로 잡는 것이 가장 자연스럽다.

좋은 어려움 예시:

- 발급 성공과 소진 후 실패 요청이 섞이며 성능 지표가 왜곡되는 문제
- 전략별 의존 인프라 차이 때문에 순수 비교가 어려운 문제
- 처리량만 보면 빨라 보여도 초과 발급/중복 발급이 없는지 같이 검증해야 하는 문제
- Step 4에서 비동기 저장으로 API 의미가 달라지는 문제
- 빠른 전략일수록 장애 대응과 운영 복잡도가 커지는 문제

즉, `어려움`은 따로 꾸미는 것이 아니라 `비교 실험을 제대로 하려고 하니 생긴 문제`로 정리하면 된다.

## Claude와 맞추면 좋은 다음 작업 우선순위

### 1순위

- Step 1 결과 기록 방식 정리
- 성능 지표와 정합성 지표를 분리해 해석 기준 명확화

### 2순위

- Step 2 구현 전에 공통 응답 계약/비교 기준 정리
- Step 4까지 같은 프로젝트 안에 둘 수 있는 기준 확정

### 3순위

- ADR 기준 재정리
- 일반 개발 규칙보다 실험/의사결정 문서에 더 집중

### 4순위

- 전략별 결과 누적 문서 템플릿 작성
- Step 2~4 결과를 같은 형식으로 기록

## 최종 제안

현재 `coupon-rush`는 좋은 방향으로 가고 있다.  
문제는 방향 자체가 아니라, 이제부터는 `많이 만드는 것`보다 `공정하게 비교하고 잘 설명되는 구조`를 만드는 것이 더 중요하다는 점이다.

따라서 다음 단계의 기준은 아래처럼 잡는 것이 좋다.

- 기능 추가보다 비교 기준 정교화
- 기술 나열보다 병목 이동의 설명력 강화
- 문서 수 증가보다 핵심 의사결정 문서 강화
- 결과 수집보다 결과 해석 기준 명확화

이렇게 가면 `coupon-rush`는 단순한 개인 프로젝트가 아니라,  
실무 경험을 확장해 동시성 전략의 성능/정합성/운영 복잡도를 검증한 프로젝트로 설명할 수 있다.
