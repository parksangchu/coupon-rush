# 부하 테스트 인프라 구성

| 항목 | 내용 |
|------|------|
| 상태 | resolved |
| 최종 수정 | 2026-03-16 |

## 배경

4가지 동시성 전략을 k6 부하 테스트로 비교해야 한다. 부하 테스트의 목표 RPS 설정 방식과 k6/모니터링의 물리적 배치를 결정해야 했다.

## 타임라인

### 2026-03-16: 고정 RPS 목표 → 점진적 증가 방식으로 전환

**문제/질문**: 부하 테스트 목표 RPS를 얼마로 잡을 것인가?

**검토한 선택지**:
1. 고정 RPS 목표 (50,000 / 5,000) — 인프라 스펙에 따라 도달 가능 여부가 달라져 설계와 실행이 불일치
2. 점진적 증가 (500 → 최대 10,000) — 각 전략이 꺾이는 지점을 찾는 방식. 인프라 스펙과 무관하게 전략 간 상대적 차이를 비교 가능

**결정과 이유**: 500 RPS에서 시작해 전략 간 차이가 나타날 때까지 점진적으로 올린다. 상한 10,000 RPS. 처음에 50,000 → 5,000으로 하향했으나, t3.micro에서 Spring Boot 벤치마크가 ~400 RPS 수준이라 고정 목표 자체가 인프라 변수에 휘둘렸다. "어디서 꺾이는가"를 찾는 방식이 프로젝트 원칙("실측 기반 판단")에 부합한다.

**결과**: k6에서 rate를 환경변수(`__ENV.RATE || '500'`)로 제어. 차이가 안 나면 인스턴스 스펙을 올린다 (Terraform으로 변경 후 재실험).

### 2026-03-16: k6 + 모니터링을 같은 EC2에 통합 배치

**문제/질문**: k6 부하 생성기와 모니터링(Prometheus + Grafana)을 어디에 배치할 것인가?

**검토한 선택지**:
1. 로컬에서 k6 + 모니터링 실행 — 메모리 부담(5~7GB), 인터넷 jitter, 고RPS에서 로컬이 병목
2. 앱 서버에 모니터링 통합 — 모니터링이 앱 성능에 영향 → 측정 결과 왜곡
3. k6 EC2에 모니터링 통합 — 로컬은 브라우저만, VPC 내부 통신(<1ms)으로 안정적

**결정과 이유**: 선택지 3 채택. k6와 모니터링을 같은 EC2에 배치. 인스턴스를 t3.small(2GB)로 업그레이드, 디스크 20GB. 리소스 이름을 `k6` → `test`로 변경.

**결과**:
- 로컬 메모리 사용량: ~5GB → 브라우저만(~1GB)
- k6→app 네트워크 레이턴시: 인터넷 경유(10~50ms jitter) → VPC 내부(<1ms)
- 월 비용 증가: 약 +$19/월 (테스트 시에만 켜두면 무시 가능)
- terraform apply마다 app 서버 IP가 바뀔 수 있어 prometheus-aws.yml 타겟 IP 수동 교체 필요

## 현재 결론

부하 테스트는 점진적 RPS 증가 방식으로, k6+모니터링은 test EC2(t3.small)에 통합 배치. 전략별로 "어디서 꺾이는가"를 찾아 상대적 차이를 비교한다.
