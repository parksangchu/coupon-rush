## 배경

4가지 동시성 전략을 k6 부하 테스트로 비교한다. 테스트 결과가 유의미하려면 인프라 배치, 부하 수준, 측정 항목의 기준이 명확해야 한다. 초기 설계부터 첫 AWS 테스트 결과를 거치며 기준을 구체화해왔다.

## 타임라인

### 2026-03-16: 고정 RPS 목표 → 점진적 증가 방식으로 전환

**문제/질문**: 부하 테스트 목표 RPS를 얼마로 잡을 것인가?

**검토한 선택지**:
1. 고정 RPS 목표 (50,000 / 5,000) — 인프라 스펙에 따라 도달 가능 여부가 달라져 설계와 실행이 불일치
2. 점진적 증가 (500 → 최대 10,000) — 각 전략이 꺾이는 지점을 찾는 방식. 인프라 스펙과 무관하게 전략 간 상대적 차이를 비교 가능

**결정과 이유**: 500 RPS에서 시작해 전략 간 차이가 나타날 때까지 점진적으로 올린다. 상한 10,000 RPS. 처음에 50,000 → 5,000으로 하향했으나, t3.micro에서 Spring Boot 벤치마크가 ~400 RPS 수준이라 고정 목표 자체가 인프라 변수에 휘둘렸다. "어디서 꺾이는가"를 찾는 방식이 프로젝트 원칙("실측 기반 판단")에 부합한다.

**결과**: k6에서 rate를 환경변수(`__ENV.RATE || '500'`)로 제어. 차이가 안 나면 인스턴스 스펙을 올린다 (Terraform으로 변경 후 재실험).

### 2026-03-16: k6 + 모니터링을 같은 EC2에 통합 배치

**문제/질문**: k6 부하 생성기와 모니터링(Prometheus + Grafana)을 어디에 배치할 것인가?

**검토한 선택지**:
1. 로컬에서 k6 + 모니터링 실행 — 메모리 부담(5~7GB), 인터넷 jitter, 고RPS에서 로컬이 병목
2. 앱 서버에 모니터링 통합 — 모니터링이 앱 성능에 영향 → 측정 결과 왜곡
3. k6 EC2에 모니터링 통합 — 로컬은 브라우저만, VPC 내부 통신(<1ms)으로 안정적

**결정과 이유**: 선택지 3 채택. k6와 모니터링을 같은 EC2에 배치. 인스턴스를 t3.small(2GB)로 업그레이드. 리소스 이름을 `k6` → `test`로 변경.

**결과**:
- 로컬 메모리 사용량: ~5GB → 브라우저만(~1GB)
- k6→app 네트워크 레이턴시: 인터넷 경유(10~50ms jitter) → VPC 내부(<1ms)
- 월 비용 증가: 약 +$19/월 (테스트 시에만 켜두면 무시 가능)
- terraform apply마다 app 서버 IP가 바뀔 수 있어 prometheus-aws.yml 타겟 IP 수동 교체 필요

### 2026-03-19: 쿠폰 수량 기준 — 100개 vs 10,000개

**문제/질문**: 설계문서에 쿠폰 10,000개로 명시되어 있는데, 첫 테스트는 k6 기본값 100개로 실행됐다. 쿠폰 수량을 정하는 기준이 뭔가?

**검토한 선택지**:

쿠폰 수량은 경합의 강도를 바꾸지 않는다. 500 RPS면 쿠폰이 100개든 10,000개든 초당 500개 요청이 같은 row의 lock을 경쟁한다. 쿠폰 수량이 바꾸는 것은 경합이 얼마나 오래 지속되는가뿐이다.

| 쿠폰 수 | 소진 시점 (500 RPS) | 발급 구간 | 거절 구간 | 측정 초점 |
|---------|-------------------|----------|----------|----------|
| 100개 | ~0.2초 | 극히 짧음 | ~29.8초 | 거절 성능 위주 |
| 1,000개 | ~2초 | ~2초 | ~28초 | 여전히 거절 위주 |
| 10,000개 | ~20초 | ~20초 | ~10초 | 발급 중 경합 + 거절 모두 관찰 |
| 15,000개 | 미소진 | 30초 전체 | 없음 | 발급만 관찰 |

**결정과 이유**: 10,000개. 500 RPS × 30초에서 발급 구간(~20초)과 거절 구간(~10초)이 모두 충분히 포함된다. 단, Pessimistic Lock처럼 처리량이 낮은 전략은 30초 내에 10,000개를 소진하지 못할 수 있다 — 이 자체가 전략의 한계를 보여주는 유의미한 결과다.

**결과**: k6 시나리오 기본값을 100 → 10,000으로 변경.

### 2026-03-19: 테스트 이원화 — 정합성 검증과 성능 비교 분리

**문제/질문**: 10,000개로 테스트하면 느린 전략은 쿠폰을 소진하지 못한다. 소진이 안 되면 정합성(초과발급 없음)을 완전히 검증할 수 없다. 전략마다 쿠폰 수를 다르게 하면 조건이 불공정해진다.

**검토한 선택지**:
1. 쿠폰 수량을 전략별로 조정 — 조건이 달라져 비교 불공정
2. gracefulStop을 충분히 늘려 모든 요청이 완료될 때까지 대기 — Pessimistic Lock에서 총 75-100초 소요, 마지막 요청의 응답 시간 60초+ → 비현실적
3. 목적별로 테스트를 두 가지로 분리 — 정합성은 소진 가능한 수량으로, 성능은 동일 조건으로

**결정과 이유**: 선택지 3 채택.

- **테스트 1 (정합성 검증)**: 쿠폰 1,000개 / 500 RPS / 30초. 모든 전략이 30초 내에 소진 가능한 수량. `issued == total` 확인으로 초과발급 없음을 완전히 검증.
- **테스트 2 (성능 비교)**: 쿠폰 10,000개 / 500 RPS / 30초. 전략 간 처리량, 응답속도, 병목 비교. 소진 못해도 OK — dropped_iterations, 실제 발급 수 자체가 비교 메트릭.

**결과**: 모든 전략에서 테스트 1, 2를 순서대로 수행. 인프라는 한 번만 올리고 k6만 두 번 실행.

### 2026-03-19: RPS 증가 전략

**문제/질문**: 500 RPS에서 시작한 후, 언제 RPS를 올리는가? 올리면 모든 전략을 다시 테스트하는가?

**검토한 선택지**:
1. 모든 전략을 매 RPS 단계에서 테스트 — 이미 한계가 드러난 전략도 반복 테스트해야 함
2. 차이가 안 나는 전략 간에만 RPS 증가 — 한계가 명확한 전략은 재테스트 불필요

**결정과 이유**: 선택지 2 채택. Pessimistic Lock이 500 RPS에서 이미 10,000개도 소진 못하면, 1,000 RPS에서 재테스트할 이유가 없다. RPS 증가는 비슷한 성능을 보이는 전략들의 차이를 찾기 위한 것이다. 증가 단계: 500 → 1,000 → 2,000 → 5,000 → 10,000 RPS.

**결과**: 전략별로 필요한 RPS 단계만 테스트. 설계문서의 점진적 증가 원칙을 유지하되, 불필요한 반복을 제거.

## 현재 결론

부하 테스트는 목적별로 이원화한다:
- **정합성 검증**: 쿠폰 1,000개, 소진 완료 확인
- **성능 비교**: 쿠폰 10,000개, 동일 조건에서 처리량/응답속도/병목 비교

k6 + 모니터링(Prometheus + Grafana)은 test EC2(t3.small)에 통합 배치. RPS는 500에서 시작하고, 차이가 안 나는 전략 간에만 점진적으로 증가한다. 비교 메트릭: 실제 발급 수, dropped_iterations, duration_issued/rejected p50/p95/p99, 모니터링 데이터(connection pool, CPU).
