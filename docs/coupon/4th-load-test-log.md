# 4차 부하 테스트 기록 (maxVUs 10,000 + p99)

| 항목 | 내용 |
|------|------|
| 상태 | resolved |
| 최종 수정 | 2026-03-26 |

## 배경

3차 테스트(maxVUs 5,000)에서 두 가지 문제 발견:
1. Single UPDATE, Redis Counter는 maxVUs 2,000에서 측정 → 실제 VU 미확인
2. p99 미수집 → Kafka/Redis Streams VU 차이의 근거 부족

4개 전략을 maxVUs 10,000 + p99 포함 조건으로 재측정한다.

## 테스트 조건

- 인프라: App EC2 m6i.large, RDS db.m6g.large, ElastiCache cache.m6g.large, Test EC2 m6i.large, Kafka EC2 m6i.large
- k6 maxVUs: 10,000
- summaryTrendStats: avg, min, med, max, p(90), p(95), p(99), count
- handleSummary: JSON 저장
- 쿠폰 10,000개, 30초
- JVM 웜업: 1,000 RPS x 2회

### 본측정

| RPS | p95 | p99 | maxVUs | dropped | issued count | 비고 |
|-----|-----|-----|--------|---------|------------|------|
| 100 | 8.7ms | 13.8ms | 2 | 0 | 3,001 | 쿠폰 미소진 (3,001/10,000) |
| 200 | 96ms | 127ms | 100 | 0 | 6,000 | 쿠폰 미소진 |
| 300 | 22ms | 55ms | 100 | 0 | 9,001 | 쿠폰 미소진 |
| 400 | 201ms | 241ms | 100 | 0 | 10,000 | 쿠폰 소진, VU 여유 |
| 500 | 2,085ms | 2,213ms | 1,019 | 944 | 10,000 | **급격한 악화** |
| 1,000 | 8,709ms | 9,082ms | 6,126 | 7,920 | 10,000 | |
| 2,000 | 12,479ms | 12,963ms | 10,000 (포화) | 33,210 | 10,000 | VU 상한 도달 |
| 3,000 | 12,976ms | 13,534ms | 10,000 (포화) | 62,285 | 10,000 | 2,000과 비슷 — 이미 천장 |

### 분석

400 RPS까지는 p95 200ms 이하, VU 100 이내로 안정적. **500 RPS에서 급격히 붕괴** (p95 200ms → 2,085ms, VU 100 → 1,019).

이전 3차 테스트에서 "1,000 RPS p95 110ms"는 재현 불가. 400 RPS에서 이미 200ms인데 1,000에서 110ms가 나올 수 없다. **3차의 Single UPDATE 1,000 RPS 결과(110ms)는 오류로 판단.**

재배포 후 재측정에서도 p95 8,617ms, VU 6,108으로 동일 결과 재현 — 오류 확정.

300 RPS에서 p95 22ms가 200 RPS의 96ms보다 빠른 이유: 300 RPS × 30s = 9,000으로 쿠폰 10,000개를 소진 못함. 전체가 발급 성공 경로(UPDATE 1 row)만 거침. 200 RPS(6,000건)도 마찬가지지만 200 RPS는 쿠폰 소진 후 거절 경로도 포함될 수 있어 편차 존재.

5,000 RPS에서 p95 2,017ms. 3차 결과(2,215ms)와 유사 — Redis Counter의 3차 결과는 대체로 정확했던 것으로 보임.

### 웜업

### 본측정

| RPS | p95 | p99 | maxVUs | dropped | issued count | 비고 |
|-----|-----|-----|--------|---------|------------|------|
| 1,000 | 1.7ms | 4.4ms | 100 | 0 | 10,000 | |
| 2,000 | 2.3ms | 9.7ms | 100 | 0 | 10,000 | |
| 5,000 | 12ms | 27ms | 344 | 1,857 | 10,000 | |
| 7,000 | 12ms | 19ms | 651 | 9,638 | 10,000 | |
| 10,000 | 17ms | 34ms | 4,843 | 93,326 | 10,000 | |

### 분석

전 구간에서 Kafka보다 p95 빠름 (1,000 RPS: 1.7ms vs 7.4ms). p99에서 차이가 더 벌어지는지 확인:

| RPS | Kafka p99 | Streams p99 | 차이 |
|-----|-----------|-------------|------|
| 1,000 | 9.3ms | 4.4ms | Streams 2x 빠름 |
| 5,000 | 22ms | 27ms | 비슷 |
| 10,000 | 33ms | 34ms | **거의 동일** |

p99에서 Kafka와 Redis Streams의 차이는 고 RPS에서 수렴한다. **p99가 VU 차이를 설명하기에는 부족**할 수 있다.

VU 비교:

| RPS | Kafka maxVUs | Streams maxVUs |
|-----|-------------|---------------|
| 5,000 | 344 | 344 |
| 7,000 | 788 | 651 |
| 10,000 | 3,806 | 4,843 |

10,000 RPS에서 **Redis Streams(4,843)가 Kafka(3,806)보다 VU가 더 높다!** 3차 결과(Kafka 5,000, Streams 3,877)와 역전. dropped가 다르기 때문으로 보임: Kafka dropped 82,603 vs Streams 93,326 — Streams가 더 많이 드롭되어 실제 처리량이 낮음.
